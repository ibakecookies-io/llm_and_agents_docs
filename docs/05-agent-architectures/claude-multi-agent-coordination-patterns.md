# Claude: Multi-agent coordination patterns

> Кратко: разбор статьи Claude о пяти паттернах координации multi-agent систем, о том, как выбирать между иерархией, автономными командами, event-driven маршрутизацией и shared state, и почему начинать стоит с самой простой архитектуры, которая покрывает задачу.

Источник: [Multi-agent coordination patterns: Five approaches and when to use them](https://claude.com/blog/multi-agent-coordination-patterns), Claude Blog, 10 Apr 2026.

## Зачем читать

- Чтобы не выбирать multi-agent архитектуру по "красоте", а выбирать ее по структуре задачи.
- Чтобы понять различия между `generator-verifier`, `orchestrator-subagent`, `agent teams`, `message bus` и `shared state`.
- Чтобы увидеть, где каждый паттерн выигрывает, а где начинает ломаться.
- Чтобы получить практическую логику эволюции архитектуры: от простого паттерна к более сложному по мере роста требований.

## Ключевая идея

Главный тезис статьи очень инженерный: **нет лучшей multi-agent архитектуры вообще, есть только архитектура, подходящая под форму информационного потока и контекстные границы задачи**.

Claude предлагает выбирать паттерн не по ролям агентов, а по трем структурным вопросам:
- как долго агент должен удерживать контекст;
- насколько подзадачи зависят друг от друга;
- должны ли агенты обмениваться событиями, результатами или общей рабочей памятью.

Отсюда и основной практический вывод: **начинать нужно с минимально сложной схемы, которая решает задачу, и усложнять архитектуру только там, где текущий паттерн становится узким местом**.

## Основные архитектурные схемы

Статья описывает пять паттернов координации.

### 1. Generator-verifier

Самый простой паттерн: один агент производит результат, другой проверяет его по явным критериям и либо принимает, либо отправляет на доработку.

```text
Task
  |
  v
[Generator] ----> draft_1 ----> [Verifier]
   ^                               |
   |                               |
   +----- feedback / rejection <---+
                    |
                    v
                accept / stop
```

#### Где работает хорошо

- quality-critical output;
- случаи, где критерии проверки можно сделать явными;
- code generation + tests;
- factual verification;
- compliance / policy checks;
- rubric-based evaluation.

#### Ключевая идея

Сила паттерна не в "двух агентах", а в **разделении генерации и оценки**. Если критерии проверки сформулированы хорошо, система получает дешевый внутренний контроль качества.

#### Где ломается

- verifier не знает, что именно проверять;
- проверка столь же сложна, как и генерация;
- цикл не сходится, generator и verifier застревают в осцилляции.

#### Практический вывод

Этот паттерн подходит, когда цена ошибки выше цены дополнительной итерации. Но без явных критериев он быстро превращается в имитацию quality control.

### 2. Orchestrator-subagent

Иерархический паттерн: ведущий агент декомпозирует задачу, делегирует bounded subtasks и затем синтезирует результаты.

```text
User task
   |
   v
[Orchestrator]
   | \
   |  \----> [Subagent A] --> result A
   |-----> [Subagent B] --> result B
   \-----> [Subagent C] --> result C
            |
            v
      synthesis by orchestrator
            |
            v
        Final output
```

#### Где работает хорошо

- декомпозиция понятна заранее;
- у подзадач четкие границы;
- результаты можно вернуть в структурированном виде;
- между подзадачами мало зависимостей.

#### Ключевая идея

Это базовый default pattern для большинства multi-agent задач. Orchestrator держит глобальную цель, а subagents изолируют локальную работу и не засоряют главный контекст.

#### Где ломается

- orchestrator становится bottleneck по информации;
- зависимость между subagents приходится вручную маршрутизировать через центр;
- без явного parallelization система платит multi-agent cost, но не получает выигрыша в скорости.

#### Практический вывод

Статья рекомендует начинать именно с этого паттерна, потому что он покрывает широкий спектр задач при сравнительно умеренной orchestration complexity.

### 3. Agent teams

Паттерн автономной команды: coordinator не создает одноразовых subagents под каждую микрозадачу, а поддерживает "живых" workers, которые долго работают над своими доменами и накапливают контекст.

```text
Backlog / task queue
        |
        v
   [Coordinator]
    /    |     \
   v     v      v
[Teammate 1] [Teammate 2] [Teammate 3]
   |            |             |
 multi-step   multi-step    multi-step
 work         work          work
   |            |             |
   +---------- completed work-+
                |
                v
         coordinator collects
```

#### Где работает хорошо

- задачи независимы друг от друга;
- работа длительная и многошаговая;
- worker выигрывает от накопленного domain context;
- нужны не одноразовые bounded subtasks, а автономное sustained execution.

#### Ключевая идея

Разница с `orchestrator-subagent` не в наличии координатора, а в **длительности жизни worker-а и сохранении его контекста между заданиями**.

#### Где ломается

- workers затрагивают общие ресурсы;
- задачи оказываются не такими независимыми, как казалось;
- coordinator сложно отслеживать частичную готовность и конфликты.

#### Практический вывод

Если subagent начинает нуждаться в долговременном состоянии и повторных циклах работы, архитектура обычно уже движется в сторону agent teams.

### 4. Message bus

Паттерн event-driven координации: агенты не вызывают друг друга напрямую, а публикуют события в общую шину; router доставляет их подписанным агентам.

```text
Event source
    |
    v
[Message Bus / Router]
   |        |         |
   v        v         v
[Agent A] [Agent B] [Agent C]
   |        |         |
   +---- publish new events ----+
                |
                v
         downstream processing
```

#### Где работает хорошо

- event-driven pipelines;
- экосистема агентов растет;
- маршрутизация зависит от типа события;
- workflow формируется динамически, а не задан жестко.

#### Ключевая идея

Этот паттерн масштабирует не только число агентов, но и **способность системы эволюционировать без постоянной переделки прямых связей между компонентами**.

#### Где ломается

- сложнее трассировать, что произошло;
- ошибки маршрутизации могут приводить к silent failure;
- semantic routing через LLM добавляет отдельный класс ошибок.

#### Практический вывод

Message bus полезен, когда логика "кто должен отреагировать" уже стала слишком сложной для одного orchestrator. Цена гибкости здесь - наблюдаемость и дебаг.

### 5. Shared state

Паттерн общей рабочей памяти: агенты координируются через общую persistent store, в которую все могут читать и писать напрямую.

```text
      seed question / dataset
               |
               v
        [Shared Store]
         ^     ^     ^
         |     |     |
   [Agent A] [Agent B] [Agent C]
         |     |     |
         +-- read / write findings --+
                    |
                    v
         convergence / stop condition
```

#### Где работает хорошо

- collaborative research;
- агенты должны быстро видеть открытия друг друга;
- работа накапливается как evolving knowledge base;
- нежелателен единый coordinator как single point of failure.

#### Ключевая идея

Здесь координация происходит не через команды и не через события, а через **общую изменяемую среду**, в которой поведение системы возникает из взаимодействий агентов со store.

#### Где ломается

- агенты дублируют работу;
- появляются contradictory updates;
- возникают reactive loops, где агенты бесконечно реагируют друг на друга;
- termination condition не задана как first-class часть архитектуры.

#### Практический вывод

Shared state особенно силен для исследовательских и аналитических сценариев, но требует дисциплины вокруг convergence, versioning и правил завершения.

## Сравнение паттернов по смыслу

### Паттерн 1: что именно является каналом координации

- `generator-verifier` -> feedback loop;
- `orchestrator-subagent` -> централизованная делегация;
- `agent teams` -> coordinator + очередь задач;
- `message bus` -> события и подписки;
- `shared state` -> общая память / общий store.

### Паттерн 2: как задаются контекстные границы

- `generator-verifier`: контекст разделен на generation context и evaluation context;
- `orchestrator-subagent`: subagent получает bounded local context, orchestrator держит глобальную цель;
- `agent teams`: каждый teammate долго удерживает свой доменный context;
- `message bus`: контекст течет через события и topic routing;
- `shared state`: контекст накапливается в общем пространстве, доступном всем агентам.

### Паттерн 3: где находится узкое место

- `generator-verifier`: качество verifier criteria;
- `orchestrator-subagent`: информационная перегрузка orchestrator;
- `agent teams`: независимость задач и конфликты по shared resources;
- `message bus`: routing correctness и observability;
- `shared state`: termination logic и дублирование работы.

## Ключевые концепции и идеи статьи

### 1. Context-centric decomposition важнее ролевой схемы

Очень сильная идея статьи: систему стоит делить не только по типам ролей, а по тому, **какой контекст нужен каждому агенту и как этот контекст должен течь между ними**.

Это меняет способ проектирования:
- сначала определяем, где должны проходить границы контекста;
- потом решаем, нужна ли иерархия, шина событий или общий store;
- и только после этого распределяем роли.

### 2. Эволюция архитектуры должна быть реакцией на реальную боль

Claude прямо рекомендует не начинать со сложных схем. Хорошая логика роста выглядит так:

```text
Simple quality loop
    -> hierarchy
    -> persistent workers
    -> event routing
    -> collaborative shared memory
```

То есть архитектура усложняется только тогда, когда предыдущий паттерн уже мешает:
- критериев проверки недостаточно;
- orchestrator перегружен;
- subagents нуждаются в долгой жизни;
- routing становится слишком ветвистым;
- агентам нужна совместная рабочая память в реальном времени.

### 3. Длина жизни агента - один из ключевых design signals

Статья хорошо разделяет два режима:
- одноразовый bounded subagent;
- persistent teammate, который живет дольше одной подзадачи.

Это важный архитектурный критерий. Если worker должен помнить предыдущие шаги, накапливать domain familiarity и улучшаться на серии assignment-ов, это уже не просто delegated subagent.

### 4. Coordination pattern определяет failure modes

Каждая схема не просто "делает систему по-другому", она создает свой класс поломок:
- loop without convergence;
- information bottleneck;
- task conflict;
- silent routing failure;
- reactive store loop.

Это полезный способ думать о выборе паттерна: выбирать нужно не только по happy path, но и по типу отказов, которые ваша система сможет диагностировать и выдерживать.

### 5. Termination condition - часть архитектуры, а не операционная мелочь

Особенно это важно для `generator-verifier` и `shared state`.

Статья подчеркивает: если не определить stop criteria заранее, multi-agent система легко начинает бесконечно тратить токены без реального движения к завершению. Поэтому лимит итераций, fallback strategy, convergence threshold и time budget должны быть предусмотрены на уровне дизайна.

## Как выбирать между паттернами

### Orchestrator-subagent vs. agent teams

Выбирайте `orchestrator-subagent`, если:
- подзадачи короткие;
- результат каждой подзадачи четко оформляется;
- worker не обязан хранить контекст между вызовами.

Выбирайте `agent teams`, если:
- подзадачи долгие и многошаговые;
- worker реально выигрывает от накопленного контекста;
- coordinator не должен каждый раз "обнулять" исполнителя.

### Orchestrator-subagent vs. message bus

Выбирайте `orchestrator-subagent`, если:
- последовательность шагов в основном известна;
- routing rules не слишком сложны;
- нужен единый центр синтеза.

Выбирайте `message bus`, если:
- workflow определяется событиями;
- появляются новые типы агентов и маршрутов;
- логика ветвления разрастается быстрее, чем разумно поддерживать в orchestrator.

### Agent teams vs. shared state

Выбирайте `agent teams`, если:
- работа разделяется по независимым partition-ам;
- агентам не нужно постоянно видеть промежуточные выводы друг друга.

Выбирайте `shared state`, если:
- открытия одного агента должны сразу влиять на направление работы другого;
- знания растут как общая evolving база;
- координация должна быть децентрализованной.

### Message bus vs. shared state

Выбирайте `message bus`, если:
- система живет как pipeline событий;
- агент реагирует на событие и передает систему дальше по цепочке.

Выбирайте `shared state`, если:
- знания накапливаются и переиспользуются многократно;
- агенты возвращаются к общей картине, а не просто реактивно обрабатывают новый event.

## Практические принципы из статьи

- Начинайте с самой простой архитектуры, которая покрывает задачу.
- Не вводите multi-agent только потому, что он выглядит "продвинутым".
- Явно проектируйте termination logic.
- Смотрите на независимость задач, а не только на число агентов.
- Отделяйте случаи, где нужен event routing, от случаев, где нужна общая память.
- Рассматривайте orchestrator как удобный default, а не как вечный центр любой системы.

## Ограничения и trade-offs

- Усложнение координации почти всегда увеличивает стоимость, задержку и число failure modes.
- Shared state и message bus дают гибкость, но усложняют reasoning about system behavior.
- Agent teams требуют особенно аккуратного partitioning задач и conflict resolution.
- Generator-verifier создает ложное чувство надежности, если verifier criteria расплывчаты.
- Orchestrator-subagent прост в старте, но при росте зависимостей может стать слишком централизованным.

## Выводы

Статья ценна тем, что переводит разговор о multi-agent architecture из абстрактной плоскости в инженерную. Она не спрашивает "какие роли придумать агентам", а спрашивает:
- как циркулирует информация;
- где должен жить контекст;
- кто принимает решения о завершении;
- какие зависимости допустимы между агентами.

Главные выводы:
- `generator-verifier` нужен для quality loops с явной проверкой;
- `orchestrator-subagent` - лучший стартовый паттерн для большинства задач;
- `agent teams` нужны, когда workers должны жить долго и сохранять накопленный контекст;
- `message bus` подходит для растущих event-driven ecosystems;
- `shared state` полезен для collaborative research и совместного наращивания knowledge base;
- выбор паттерна определяется не модой, а формой информационного потока, длительностью жизни контекста и типом зависимостей между подзадачами.
