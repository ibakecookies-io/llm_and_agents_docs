# Anthropic: Writing effective tools for AI agents

> Кратко: разбор статьи Anthropic о том, почему инструменты для AI-агентов — это не обычные API, а контракты между детерминированной системой и недетерминированным пользователем, и как проектировать такие контракты так, чтобы агент ошибался реже, тратил меньше токенов и быстрее доходил до цели.

Источник: [Writing effective tools for AI agents—using AI agents](https://www.anthropic.com/engineering/writing-tools-for-agents), Anthropic Engineering.

## Зачем читать

- Чтобы перестать думать о tool layer как о простой "обертке над backend API".
- Чтобы понять, почему плохие tools ломают даже хорошо спроектированного агента.
- Чтобы получить практические принципы для выбора tool granularity, namespacing, output shape и error behavior.
- Чтобы увидеть, как Anthropic связывает tool design с evals, prompt engineering и token economy.

## Самое главное

### 1. Tool для агента — это не обычный API

Главная идея статьи очень сильная:

> tools — это контракты между детерминированной системой и недетерминированным агентом.

Для обычного API предполагается, что человек:
- читает документацию;
- понимает домен;
- знает, когда какой endpoint уместен;
- умеет интерпретировать ошибки.

Агент так не работает. Он может:
- выбрать не тот tool;
- неверно понять назначение параметров;
- послать слишком широкий запрос;
- зациклиться на плохой стратегии;
- переиспользовать инструмент вне его intended scope.

Следствие: **инструменты для агентов нужно проектировать оборонительно и педагогично**, а не только технически корректно.

### 2. Не надо превращать каждый API endpoint в отдельный tool

Один из важнейших уроков статьи: tool layer должен следовать **пользовательским задачам и рабочим сценариям**, а не схеме базы данных или внутренней структуре backend-сервиса.

Anthropic рекомендует избегать tool sprawl:
- слишком большого числа похожих инструментов;
- мелких атомарных операций без ясных boundaries;
- нескольких способов сделать почти одно и то же.

Практический вывод:

**один хорошо спроектированный workflow-oriented tool часто лучше пяти маленьких schema-oriented tools**.

### 3. Хороший tool должен возвращать агенту осмысленный контекст

Инструмент должен возвращать не только "формально правильные данные", но и **semantic payload**, с которым модель может дальше мыслить.

То есть вместо:
- голых ID;
- cryptic status codes;
- необъяснимых внутренних полей,

лучше возвращать:
- человекочитаемые описания;
- summaries;
- подсказки к следующему действию;
- компактную, но богатую смыслом структуру.

Это один из самых практичных выводов статьи: **tool response — это часть контекста агента, а не просто транспортный пакет данных**.

### 4. Token efficiency — часть tool design, а не постфактум оптимизация

Anthropic прямо связывает дизайн tools с ценой контекста. Плохой tool:
- тащит слишком много данных;
- не умеет фильтровать;
- не умеет пагинировать;
- не умеет регулировать подробность ответа;
- заставляет модель делать лишние вызовы.

Хороший tool:
- имеет sensible defaults;
- умеет возвращать компактный ответ;
- позволяет агенту запрашивать детали только по мере необходимости;
- сообщает об ошибках так, чтобы агент мог скорректировать стратегию.

Иными словами, **token efficiency зашивается в интерфейс инструмента**.

### 5. Tool description — это prompt

Еще один ключевой тезис: описание инструмента, его имя, названия параметров и примеры использования — это не "документация в довесок", а часть prompting surface.

Если в tool description неясно:
- когда вызывать инструмент;
- какие параметры обязательны;
- в каком формате они ожидаются;
- чего ожидать в ответе,

агент будет ошибаться систематически.

Поэтому Anthropic рассматривает tool descriptions как объект prompt engineering и рекомендует улучшать их итеративно через evals.

## Ключевая архитектурная идея

Статья предлагает смотреть на tools как на **эргономический слой между агентом и внешним миром**.

### Базовая схема

```text
User intent
   |
   v
[Agent reasoning loop]
   |
   | decides which capability is needed
   v
[Tool interface]
   |
   +--> clear name / namespace
   +--> explicit description
   +--> unambiguous parameters
   +--> token-aware output shape
   +--> helpful errors and guidance
   |
   v
[Deterministic system]
   |
   +--> database
   +--> API
   +--> file system
   +--> external service
```

Смысл схемы:
- модель взаимодействует не с raw backend, а с **специально спроектированным интерфейсом**;
- этот интерфейс должен не только давать capability, но и снижать вероятность misuse;
- хороший tool направляет поведение агента почти так же сильно, как хороший prompt.

### Более прикладная интерпретация

```text
Плохой tool:
- низкоуровневый
- двусмысленный
- многословный
- возвращает сырой шум
- плохо сообщает об ошибках

Хороший tool:
- соответствует задаче пользователя
- очевиден по названию
- ограничен по области действия
- экономен по токенам
- возвращает семантически полезный ответ
- помогает агенту скорректировать следующий шаг
```

Это очень сильный mental model для любого MCP- или function-calling стека.

## Что в статье особенно ценно

### Tool ergonomics важнее формальной полноты

Очень полезная мысль статьи: инструмент может быть technically complete, но behaviorally bad for agents.

Например, можно открыть агенту:
- двадцать endpoints;
- десятки пересекающихся параметров;
- множество слабо различимых имен;

и получить систему, которая "может почти все", но делает это плохо.

Anthropic фактически говорит: **agent ergonomics важнее API completeness**.

### Namespacing — не косметика, а снижение когнитивной нагрузки

Статья подчеркивает важность согласованных префиксов и группировки related tools. Это кажется мелочью, но на практике:
- снижает ошибки выбора;
- делает capability surface понятнее;
- помогает агенту быстрее ориентироваться в tool set.

Особенно это важно, когда в системе есть несколько MCP-серверов или доменных наборов возможностей.

### Return shape влияет на reasoning quality

Один из недооцененных architectural points: формат ответа инструмента влияет не только на стоимость, но и на качество рассуждений модели.

Если вернуть:
- короткий, смысловой summary;
- понятные поля;
- полезные human-readable labels;

модель с большей вероятностью сделает правильный следующий шаг, чем если вернуть сырую техническую структуру.

### Error messages должны обучать агента, а не просто сигнализировать о сбое

Статья очень правильно подчеркивает, что для агента ошибка — это не конец, а поворотный момент в стратегии.

Поэтому сообщение вида:
- `TOO_MANY_RESULTS`

хуже, чем:
- "найдено 847 записей, сузьте диапазон дат или добавьте category filter".

Вторая версия не просто сообщает о проблеме, а **направляет следующее действие модели**.

## Необычные и интересные кейсы

### 1. Финансовый tool для расходов как пример "workflow over schema"

В пересказе статьи особенно полезен пример с expense tool. Вместо набора мелких инструментов вроде:
- получить расход по ID;
- список расходов;
- фильтр по категории;
- поиск расходов,

лучше сделать один хорошо продуманный инструмент вроде:
- `get_expenses_by_date_range`

с:
- фильтрами;
- лимитами;
- режимами ответа;
- понятными ошибками.

Это очень наглядный кейс того, что tool design нужно строить вокруг **реального сценария использования**, а не вокруг внутренней модели данных.

### 2. Configurable verbosity как встроенный control over context

Интересная идея из статьи: инструменту можно дать параметр типа `response_format`, чтобы агент сам выбирал:
- краткий ответ для быстрой проверки;
- подробный для анализа.

Это сильный паттерн, потому что:
- снижает token waste;
- не требует плодить отдельные tools;
- позволяет агенту адаптировать detail level к задаче.

### 3. Ограничение диапазона запроса — не просто guardrail, а защита reasoning budget

Пример с датами больше года полезен тем, что показывает: validation нужна не только для correctness, но и для экономии контекста и вычислительного внимания.

То есть tool guardrails должны защищать:
- backend;
- стоимость;
- контекст;
- качество agent behavior.

### 4. Tool design через eval-driven iteration

Anthropic рекомендует не пытаться "угадать идеальный tool" с первого раза, а:
- быстро собрать прототип;
- гонять realistic evals;
- смотреть, где агент путается;
- переписывать descriptions, параметры и output.

Это очень важный operational lesson: **хорошие tools проектируются не в теории, а через поведение агента на задачах**.

## Практические принципы из статьи

### Принцип 1. Проектируйте tools вокруг пользовательских задач

Правильный вопрос:

> какую задачу пользователь хочет решить?

Неправильный вопрос:

> как отразить мои внутренние API и таблицы в наборе functions?

### Принцип 2. Минимизируйте overlapping functionality

Если у агента есть несколько похожих инструментов, он будет ошибаться чаще. Поэтому:
- один domain action = один понятный tool;
- пересечения нужно сокращать;
- intent boundaries должны быть явными.

### Принцип 3. Возвращайте смысл, а не сырой шум

В tool response стоит возвращать:
- нужные названия;
- summaries;
- краткие вычисленные итоги;
- поля, которые агент может сразу использовать в reasoning.

Чем меньше промежуточных расшифровок нужно модели, тем лучше.

### Принцип 4. Делайте ответы token-aware по умолчанию

Хорошие приемы по духу статьи:
- лимиты по умолчанию;
- пагинация;
- фильтры;
- selectable verbosity;
- короткие summaries;
- отказ возвращать слишком большие наборы без уточнения.

### Принцип 5. Ошибки должны подсказывать лучший следующий шаг

Сообщение об ошибке должно отвечать не только на вопрос "что пошло не так", но и на вопрос:

> как агенту теперь действовать умнее?

### Принцип 6. Tool descriptions нужно оптимизировать как prompts

Имя, описание, параметры, примеры вызова, enum values — все это влияет на поведение модели. Это значит, что tool design нужно тестировать через:
- evals;
- реальные traces;
- наблюдение за систематическими промахами.

## Что можно забрать в собственную архитектуру

### Минимальный checklist для любого нового tool

Перед публикацией инструмента полезно проверить:

1. Понятно ли по имени, что он делает?
2. Не дублирует ли он другой tool?
3. Соответствует ли он пользовательскому workflow?
4. Ясны ли все параметры?
5. Ограничивает ли он слишком широкие запросы?
6. Может ли он вернуть compact response?
7. Подсказывает ли его ошибка следующий шаг?

### Полезный baseline

Для production-инструментов по мотивам статьи разумно иметь:

- domain prefix или namespacing;
- короткое, конкретное описание "когда использовать";
- few-shot usage hints в описании;
- ограничения на слишком широкие запросы;
- human-readable output;
- лимиты и фильтры по умолчанию.

### Когда дробить tool, а когда объединять

Объединять стоит, когда:
- это один пользовательский workflow;
- дробление создает лишнюю неоднозначность;
- агенту придется часто делать несколько очевидных вызовов подряд.

Дробить стоит, когда:
- у операций разные intent boundaries;
- разный риск;
- разные права доступа;
- разный формат результата;
- один "универсальный" tool становится слишком расплывчатым.

## Ограничения и trade-offs

- Слишком "умный" универсальный tool может стать неясным по контракту.
- Слишком компактный output может убрать важные детали.
- Слишком подробный output быстро забивает контекст.
- Сильные guardrails улучшают надежность, но иногда уменьшают гибкость.
- Проектирование tools требует evals и наблюдения за реальным agent behavior, а это отдельная инженерная работа.

## Что особенно важно не пропустить

### Хороший tool направляет поведение агента

Это, возможно, главный скрытый урок статьи. Инструмент не просто дает capability — он формирует:
- какие стратегии агент считает доступными;
- сколько шагов он делает;
- сколько контекста тратит;
- как быстро исправляется после неудачи.

### Human-friendly и agent-friendly часто совпадают

Anthropic отмечает важное совпадение: инструменты, удобные для агентов, часто оказываются удобными и для людей.

Это хороший сигнал, что принципы tool clarity и semantic ergonomics универсальны.

### Tools нельзя проектировать отдельно от prompt и evals

Tool layer — часть более широкой системы. Он связан с:
- prompting;
- context management;
- error recovery;
- evaluation loops.

Поэтому хороший tool design почти всегда возникает из совместной настройки:
- prompts;
- tools;
- traces;
- evals.

## Итог

Это одна из самых полезных статей для тех, кто уже понял, что "дать агенту tools" недостаточно. Настоящая инженерная работа начинается после этого — в том, **как именно эти tools выглядят, как объясняются и какую форму данных возвращают**.

Главные выводы:
- tools нужно проектировать под workflow, а не под schema;
- overlapping functionality нужно сокращать;
- ответы должны быть семантически полезными и экономными по токенам;
- ошибки должны направлять следующий шаг агента;
- tool descriptions — это prompt engineering surface;
- хорошие инструменты рождаются через eval-driven iteration.
