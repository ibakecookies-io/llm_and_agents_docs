# Anthropic: effective context engineering for AI agents

> Кратко: разбор статьи Anthropic о том, что для агентных систем важно не только "как написать prompt", но и как управлять всем контекстом целиком: инструкциями, инструментами, историей, внешними данными, памятью и механизмами сжатия.

Источник: [Effective context engineering for AI agents](https://www.anthropic.com/engineering/effective-context-engineering-for-ai-agents), Anthropic Engineering, 29 Sep 2025.

## Зачем читать

- Чтобы перейти от узкого prompt engineering к более полезной инженерной рамке: **управление полным состоянием контекста агента**.
- Чтобы понять, почему длинный контекст не равен хорошему контексту.
- Чтобы увидеть, как Anthropic мыслит о retrieval, памяти, compaction и sub-agent архитектурах.
- Чтобы забрать практические принципы для систем, которые работают долго, используют tools и регулярно выходят за рамки одного окна контекста.

## Самое главное

### 1. Context engineering шире prompt engineering

Главная идея статьи: по мере роста автономности агентов инженерная задача смещается от "какие слова положить в system prompt" к вопросу:

> какая конфигурация контекста с наибольшей вероятностью приведет модель к нужному поведению?

Под контекстом понимается не только prompt, но и:
- system instructions;
- tools и их описания;
- few-shot примеры;
- message history;
- данные из внешних источников;
- артефакты памяти и заметок;
- все, что агент подгружает в runtime.

Это важный сдвиг мышления: **агент проектируется не только через инструкции, но и через архитектуру доступа к информации**.

### 2. Контекст — конечный ресурс, а не бесплатное хранилище

Anthropic подчеркивает, что контекст нужно воспринимать как **finite attention budget**. Даже если модель поддерживает длинные окна, полезность дополнительных токенов убывает.

Ключевые причины:
- возникает `context rot`: по мере роста контекста ухудшается точность извлечения и удержания сведений;
- у трансформеров внимание распределяется по множеству pairwise-связей между токенами;
- длинные последовательности архитектурно тяжелы и менее "естественны" для моделей, чем более короткие.

Вывод жесткий и практический:

**больше контекста не означает лучшее качество**. Часто лучше дать меньше, но более сигнализированных токенов.

### 3. Главная задача — найти минимальный набор high-signal токенов

Статья несколько раз возвращается к одной и той же идее:

> нужно найти наименьший набор контекста, который максимизирует шанс нужного поведения.

Это относится ко всему:
- к prompt-ам;
- к tool descriptions;
- к примерам;
- к истории сообщений;
- к runtime retrieval.

Это, пожалуй, основной operational principle всей статьи.

### 4. Хороший агент не тащит все в контекст сразу, а подгружает нужное "just in time"

Одна из сильнейших идей статьи — смещение от наивного preloading к **dynamic retrieval during execution**.

Вместо того чтобы сразу запихнуть в контекст весь "возможно полезный" материал, агент:
- хранит lightweight references;
- ориентируется по именам файлов, путям, timestamps, структуре каталогов;
- сам вызывает инструменты, чтобы подгрузить нужное по мере работы.

Это важный архитектурный поворот:
- меньше контекстного шума;
- меньше stale data;
- больше автономной навигации;
- лучше progressive disclosure информации.

### 5. Long-horizon agents требуют специальных механизмов

Для задач, которые идут долго и неизбежно переполняют контекст, статья выделяет три главные техники:

- **compaction** — сжатие истории и перезапуск работы на summary;
- **structured note-taking / memory** — внешняя память через заметки и артефакты;
- **sub-agent architectures** — вынос глубоких исследований и поиска в отдельные подагенты с чистыми контекстами.

Это не "дополнительные оптимизации", а фактически **базовые архитектурные приемы** для серьезных агентных систем.

## Ключевая архитектурная идея

Статья предлагает смотреть на агент как на систему, которая **не просто получает prompt, а постоянно пересобирает свой рабочий контекст**.

### Базовая схема context engineering

```text
                 +----------------------+
                 | System instructions  |
                 +----------------------+
                            |
                            v
                 +----------------------+
                 |  Minimal tool set    |
                 |  + tool guidance     |
                 +----------------------+
                            |
                            v
User task --> [Agent loop / reasoning] <--> [Runtime retrieval tools]
                            |                     |
                            |                     +--> files / DB / web / notes
                            |
                            +--> [Message history]
                            |
                            +--> [Few-shot examples]
                            |
                            +--> [External memory / notes]
                            |
                            +--> [Compaction / context reset]
                            |
                            +--> [Sub-agents for deep exploration]
```

Смысл схемы:
- агент не должен держать "все знание" в окне контекста;
- часть контекста задается заранее;
- часть извлекается во время работы;
- часть сжимается;
- часть выносится во внешнюю память;
- часть делегируется подагентам.

### Более практичная интерпретация

```text
Static context:
- system prompt
- canonical examples
- tool descriptions

Dynamic context:
- только что найденные файлы
- результаты конкретных запросов
- рабочие заметки
- недавняя история

Pressure relief:
- compaction
- clearing old tool outputs
- sub-agent delegation
```

Это хороший mental model для проектирования агентных систем: **думать не "что еще добавить в prompt", а "какая часть информации должна быть статической, какая динамической, какая внешней, а какая вообще не должна попадать в основной контекст"**.

## Что в статье особенно ценно

### Prompt altitude: слишком конкретно и слишком расплывчато — оба варианта плохи

Anthropic описывает "правильную высоту" system prompt как Goldilocks zone между двумя плохими крайностями:

- слишком жесткий, хрупкий prompt с зашитой if-else логикой;
- слишком общий prompt, который ничего толком не задает.

Это очень полезное замечание. Хороший системный prompt:
- достаточно конкретен, чтобы направлять поведение;
- но не превращается в хрупкую программу, замаскированную под prose.

Иначе вы или переспецифицируете модель, или оставите ее без операционных ориентиров.

### Tools — это не просто capability layer, а контракт с информационным пространством

Статья хорошо показывает, что tools определяют:
- как агент может действовать;
- как он может получать новый контекст;
- насколько эффективно он расходует токены.

Очень сильный вывод:

**плохой набор инструментов ломает не только execution, но и context strategy**.

Если tools:
- дублируют друг друга;
- создают неоднозначность выбора;
- возвращают слишком много данных;
- плохо названы и плохо типизированы,

то агент начинает тратить и контекст, и reasoning budget впустую.

### Примеры важнее длинных списков правил

Anthropic прямо не советует забивать prompt списками edge cases. Вместо этого они рекомендуют давать:
- немного;
- но хорошо подобранных;
- canonical и разнообразных примеров.

Полезная мысль: для LLM пример часто работает лучше, чем длинное текстовое правило. Это особенно важно в агентных системах, где prompt и так быстро разрастается.

## Необычные и интересные кейсы

### 1. Claude Code анализирует большие базы данных, не загружая их целиком в контекст

Очень показательный пример из статьи: агент не тянет огромные объекты данных в окно контекста "на всякий случай". Вместо этого он:
- пишет точечные запросы;
- сохраняет результаты;
- использует `head` и `tail`;
- анализирует только нужные фрагменты.

Это хороший инженерный урок:

**правильный агент похож не на систему, которая все запоминает, а на систему, которая умеет быстро и осмысленно искать**.

### 2. Семантика файловой системы сама по себе становится частью контекста

Очень интересный момент: статья подчеркивает, что агент может использовать не только содержимое файлов, но и **метаданные среды**:
- путь к файлу;
- имя файла;
- расположение в иерархии;
- размер;
- timestamp.

Пример с `test_utils.py` в `tests/` против `src/core_logic/` отлично иллюстрирует, что контекстом является не только текст, но и **структура мира, в котором агент работает**.

Это важное architectural observation: хорошо организованная среда сама снижает когнитивную нагрузку модели.

### 3. Progressive disclosure вместо upfront ingestion

Еще одна сильная идея: агент не обязан знать все сразу. Он может:
- исследовать среду шаг за шагом;
- извлекать контекст слоями;
- обновлять картину мира по мере продвижения.

Это почти буквальная имитация человеческой работы:
- сначала обзор;
- затем точечное чтение;
- затем уточнение;
- потом синтез.

Такой подход особенно полезен в больших кодовых базах и исследовательских задачах.

### 4. Pokemon-кейс как пример естественного возникновения памяти

Статья приводит кейс `Claude Plays Pokemon`, где агент:
- ведет заметки о том, где был;
- отслеживает цели;
- накапливает стратегические наблюдения;
- продолжает длинные последовательности действий после context reset-ов.

Этот пример интересен тем, что память там не выглядит как "специальная ML-фича". Она ближе к рабочему журналу или полевым заметкам.

Главный вывод:

**простая внешняя память может радикально увеличить coherence даже без сложной архитектуры памяти**.

### 5. Compaction с сохранением последних файлов

У Anthropic в Claude Code compaction делается не как "сжать все подряд", а как высокоточная передача:
- архитектурных решений;
- открытых багов;
- деталей реализации;
- плюс сохранение пяти последних файлов.

Это интересный инженерный нюанс: continuity поддерживается не только summary, но и через **локально релевантные рабочие артефакты**.

### 6. Sub-agents как механизм изоляции search context

Очень ценный architectural point: sub-agents нужны не только ради параллелизма, но и ради **изоляции контекстного шума**.

Подагент может:
- потратить десятки тысяч токенов на поиск и исследование;
- а вернуть главному агенту только короткое distilled summary.

Это превращает multi-agent design в инструмент **контекстной компрессии через разделение труда**.

## Практические принципы из статьи

### Принцип 1. Думайте не prompt-ами, а контекстными состояниями

На каждом шаге задавайте вопрос:

- что именно агент видит сейчас;
- какие сигналы в этом состоянии сильные;
- какие токены лишние;
- какие данные лучше не тащить заранее.

Это более полезная рамка, чем постоянные попытки "еще чуть улучшить system prompt".

### Принцип 2. Минимизируйте контекст, не теряя поведенческого сигнала

Хороший контекст:
- не обязательно короткий;
- но обязательно плотный по смыслу.

Anthropic не продвигает наивный minimalism. Идея в другом: **убирать лишнее, но сохранять все, что реально влияет на нужное поведение**.

### Принцип 3. Инструменты должны быть token-efficient и поведенчески однозначными

Если у агента есть tools, они должны:
- иметь ясные boundaries;
- быть самодостаточными;
- возвращать компактные результаты;
- не конкурировать двусмысленно друг с другом.

Иначе tool layer сам становится источником context pollution.

### Принцип 4. Сначала используйте simplest retrieval that works

Статья очень прагматична:
- не нужно сразу строить сложнейший retrieval stack;
- сначала стоит проверить, справится ли агент с простыми references и JIT exploration;
- только потом добавлять более сложную orchestration.

Это хорошо согласуется с более общим принципом Anthropic: **do the simplest thing that works**.

### Принцип 5. Для long-horizon задач заранее проектируйте pressure-release mechanisms

Если задача долгая, нужно заранее понимать:
- как будет происходить compaction;
- где будут жить заметки;
- что будет записываться во внешнюю память;
- когда стоит делегировать подагенту.

Иначе агент рано или поздно утонет в собственной истории.

### Принцип 6. Compaction настраивается как retrieval system: сначала recall, потом precision

Это один из самых практичных советов статьи. Anthropic рекомендует:

1. сначала сделать compaction максимально "жадным" к важной информации;
2. убедиться, что ничего критичного не теряется;
3. уже потом сокращать лишнее и повышать precision.

Это редкий, но очень полезный operational detail.

### Принцип 7. Иногда достаточно просто очищать старые tool outputs

Статья отдельно говорит, что один из самых безопасных способов облегчить контекст — **tool result clearing**.

Это очень здравая мысль: если старый raw output уже выполнил свою функцию, нет причин держать его в рабочем окне.

## Что можно забрать в собственную архитектуру

### Минимальный базовый набор

Для большинства прикладных агентных систем из статьи можно вынести такой baseline:

1. **Короткий, четкий system prompt** с правильной "высотой".
2. **Небольшой набор хорошо различимых tools**.
3. **2-5 canonical few-shot примеров**, а не энциклопедию edge cases.
4. **JIT retrieval**, основанный на ссылках, путях, запросах и runtime exploration.
5. **Легкая внешняя память**: TODO, NOTES, state files.
6. **Compaction strategy** для длинных задач.

### Когда нужен гибридный подход

Anthropic считает особенно полезной hybrid strategy:
- часть контекста грузится upfront;
- остальное агент добывает сам по мере работы.

Это хорошо подходит, когда:
- есть небольшая доля почти всегда релевантной информации;
- остальная информация слишком велика или слишком динамична;
- важно избежать stale indexing.

### Когда нужны sub-agents

Sub-agents особенно полезны, когда:
- требуется параллельное исследование;
- поиск широк и шумен;
- главному агенту важно сохранить чистый reasoning context;
- нужен condensed output вместо полного журнала поиска.

## Ограничения и trade-offs

- JIT retrieval замедляет работу по сравнению с upfront retrieval.
- Агент может блуждать по среде и тратить шаги впустую, если tools или heuristics плохие.
- Слишком агрессивный compaction может удалить скрыто важные детали.
- Structured memory тоже требует дизайна: что писать, когда писать и как читать обратно.
- Multi-agent архитектуры уменьшают контекстный шум, но увеличивают orchestration complexity.
- Большой контекст все равно иногда нужен; проблема не решается одним только минимализмом.

## Что особенно важно не пропустить

### Context engineering — это про runtime, а не только про initial setup

Очень важный сдвиг в мышлении: контекст нужно проектировать **на каждом ходе агента**, а не только в момент написания system prompt.

Именно это отличает context engineering от классического prompt engineering.

### Хорошо спроектированная среда сама работает как подсказка для агента

Файловая и информационная среда может быть:
- ясной;
- индексируемой;
- семантически читаемой;
- богатой метаданными.

Это превращает environment design в часть agent design.

### Multi-agent — это не только про роли, но и про раздельные контекстные окна

Очень полезно помнить: sub-agent architecture ценна не только потому, что можно распределять задачи, но и потому что **каждый агент получает свой чистый context window**.

Это один из лучших способов бороться с context pollution.

## Итог

Это одна из самых полезных статей про агентные системы, потому что она переводит разговор из плоскости "как лучше писать prompts" в более зрелую плоскость:

**как организовать ограниченное внимание модели так, чтобы она стабильно делала нужную работу**.

Главные выводы:
- контекст — конечный ресурс;
- лучший контекст не максимальный, а наиболее сигнализированный;
- retrieval должен быть все более runtime-ориентированным;
- long-horizon задачи требуют compaction, memory или sub-agent patterns;
- архитектура информационной среды влияет на качество агента не меньше, чем prompt.
